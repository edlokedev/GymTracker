## Proposed CLAUDE.md Additions

### New (from today's activity)

- [ ] **Object identity from adapter hooks must be memoized (NEW — from c748069, production incident)** — `useAuth()` called `adaptUser()` on every render, building a fresh object each time. Consumers listing `user` in a dep array (e.g., `ExerciseBrowser`) re-ran every render, driving an infinite edge-request loop (~280k Vercel function invocations/day, ~78% of Hobby cap). Fix: `useMemo(() => adaptUser(supa.user), [supa.user])`. **Rule**: Any hook that calls an adapter or constructor function to transform context data must memoize its output on the stable source reference. Check: if the hook body contains `adaptX(y)` or `new X(y.field)` where `y` is a context/provider value, wrap with `useMemo`.

- [ ] **Client-side filter short-circuit pattern (NEW — ADR-0005)** — When the full dataset is already loaded in memory (e.g., all favourites fetched on mount), `runSearch` can short-circuit the API call and serve from in-memory state. Shape: `if (filters.favourites) { setExercises(state.favoriteExercises); setTotal(n); setHasMore(false); return; }`. Applicable only when: client has the full set (no pagination needed on the dataset), and the filter is a view over that set, not a new query. Document the revisit condition: if the dataset becomes paginated or unbounded, move server-side.

- [ ] **`reflectFavouritesIfActive` — mutation-aware filter view sync (NEW — from PR #9)** — When a collection drives a client-side filter view, every mutation path (toggle success, refresh) must call a `reflectXIfActive(newSet)` helper to update the displayed list, not just the underlying data. Without this, the filter view goes stale after a mutation. Pattern: one helper, called from every mutation path that touches the filtered collection. Test: assert that toggling off while filter is active removes the item from the displayed grid.

- [ ] **Global disable-while-any-pending for concurrent mutation protection (NEW — from PR #9)** — When a component has a single `togglingId: string | null` (not a `Set`), disable ALL instances of the interactive control when `togglingId != null`, not just the active one. This closes the concurrent different-item race (tap A then B fast) without needing to harden the hook into a full queue. Rule: `disabled={togglingId != null}` on every control, `aria-busy={togglingId === thisId}` on the one being toggled. Test must prove a second tap on a *different* item is blocked while a toggle is in flight.

- [ ] **Mode-gating auth features in reused components (NEW — from PR #9)** — When a component is reused in multiple contexts (e.g., standalone browser vs in-workout picker), gate auth-dependent UI wiring behind a mode check so that secondary usage contexts don't silently gain new behaviour. Pattern: `!onSelectExercise` (absence of picker prop) = standalone mode; only wire stars, mutations, and auth-gated controls in standalone mode. Document the mode-check condition in the plan before coding.

- [ ] **`vi.hoisted` mutable state for per-test auth control (NEW — from `favourite-star.test.tsx`)** — Use `const authState = vi.hoisted(() => ({ user: null as { id: string } | null }))` to create a shared mutable state that can be mutated per-test inside the `vi.mock('@/lib/auth', ...)` factory. Then `authState.user = { id: 'u1' }` in a test before `render`. Cleaner than re-mocking per test. Works because `vi.hoisted` runs before module evaluation.

- [ ] **`pendingMutation: true` fetch stub for in-flight UI assertions (NEW — from `favourite-star.test.tsx`)** — When testing disabled/busy state during a pending mutation, pass a flag to the fetch stub that returns a never-resolving promise for the mutation endpoint. The test renders, fires the action, then synchronously asserts disabled state before the promise resolves. Pattern: `installFetch({ pendingMutation: true })` → `fetch()` for POST/DELETE returns `new Promise(() => {})`.

- [ ] **Vercel Analytics + Speed Insights placement (NEW — from a9e3dbc)** — Mount `<Analytics />` and `<SpeedInsights />` from `@vercel/analytics/react` and `@vercel/speed-insights/react` in `__root.tsx` body, **outside** `AuthProvider`, so unauthenticated and login routes are also tracked. Use the framework-agnostic `/react` exports (not a TanStack Start preset). One-time setup; do not add per-route.
